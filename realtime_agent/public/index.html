Quickstart

  1. Copy your API key above.
  2. Make a test request to GET /tickets with the bearer header.
  3. Receive ticket data — or poll on a 3-second interval to surface new tickets in your own dashboard.
curl 
# List tickets
curl -H "Authorization: Bearer YOUR_API_KEY" \
     $BASE_URL/tickets
node 
const resp = await fetch("BASE_URL" + "/tickets?priority=high", {
  headers: { Authorization: `Bearer ${process.env.REYA_API_KEY}` },
});
const { tickets, counts } = await resp.json();
console.log(`${counts.high} high-priority tickets`);

Authentication

Every request must include an Authorization header. Two modes are accepted:

  • Bearer token — for programmatic / server-to-server use. Recommended for integrations.
  • HTTP Basic — used by the browser-facing console UI. Don't use this from your backend.
http 
Authorization: Bearer YOUR_API_KEY

REST API

All endpoints return JSON. Errors return { "error": "message" } with a non-200 status.

Tickets

MethodPathDescription
GET/ticketsList tickets. Filters: ?status=, ?priority=, ?retailer=, ?q=.
GET/tickets/:idGet a single ticket plus its linked transcript.
POST/ticketsCreate a ticket from outside the voice flow.
PATCH/tickets/:idUpdate status / priority. Approve, deny, escalate, override.
POST/tickets/:id/replyAppend a human-agent reply / internal note to the audit log.
POST/tickets/:id/assistGenerate an AI-drafted reply using transcript + customer context.
curl 
# Approve a ticket
curl -X PATCH $BASE_URL/tickets/RC-001 \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"status":"resolved","action":"approve","note":"Refund granted"}'

🔌 Customer database integration

The agent calls a lookup_customer_data tool whenever the customer asks about their own order, refund, or account. The server proxies that call to your own backend via the CUSTOMER_DATA_WEBHOOK environment variable.

This is the plug-and-play extension point. Point it at your DB API and the agent immediately knows about every order in your system.

MethodPathDescription
POST/customer/queryProxies to CUSTOMER_DATA_WEBHOOK. Body: { query_type, retailer, customer_id, order_id? }.

Your webhook receives

json 
{
  "query_type":   "order_status",    // or order_history, refund_status, shipping_status, account_info
  "retailer":     "amazon",
  "customer_id":  "customer@email.com",
  "order_id":     "112-8847431-1922"
}

Your webhook returns (your schema, anything JSON)

json 
{
  "order_id":         "112-8847431-1922",
  "status":           "in_transit",
  "carrier":          "UPS",
  "estimated_arrival": "2026-05-15",
  "items":            [{ "sku": "B07XJ8C8F5", "qty": 1 }]
}

Reya speaks the data back to the customer in plain English, applies the same guardrails (no PII leakage, no fabrication), and files a ticket if anything goes wrong.

🛡️ Guardrails

Built into the system prompt and enforced on every turn — non-negotiable, override every other instruction:

  • No PII collection — never asks for credit card, SSN, password, bank account, full DOB. Refuses if volunteered.
  • No policy hallucination — only states facts from the FAQ knowledge base or the cross-retailer reference. Three-tier confidence framework (FAQ → confident inference with hedge → honest "I don't know").
  • No binding commitments — cannot promise refunds, approve returns, or override policy. Only files structured tickets.
  • Out-of-scope refusal — declines medical, legal, financial advice and any topic outside customer support.
  • Prompt-injection resistance — recognizes "ignore previous instructions" and similar attacks; stays in support role.
  • Data minimization — only collects what's needed (name for tickets, order ID for lookups). Doesn't repeat back sensitive details.

🇪🇺 GDPR-aligned privacy controls

Built-in features for European data protection compliance:

RightHow
ACCESSFull transcript and ticket history per session via GET /tickets/:id and GET /transcripts/:sessionId.
ERASUREOne-call deletion of all session data: DELETE /sessions/:sessionId removes the transcript + any linked tickets.
AUDITEvery status change, agent reply, and AI action is logged on the ticket with timestamp + actor.
RETENTIONAuto-purge of stale data via DATA_RETENTION_DAYS env var (default 90). Runs daily.
DISCLOSUREReya opens every call with: "This is Reya, an AI support assistant. This call may be recorded for service improvement and AI training."

Embed the voice widget beta

Drop the iframe on any page you own. Audio streams directly between the browser and OpenAI — your server only handles session minting and ticket creation.

html 
<iframe
  src="BASE_URL/?token=YOUR_API_KEY"
  allow="microphone; autoplay"
  style="border:0;width:100%;height:560px"
></iframe>
