Reya is a voice support agent for US e-commerce shoppers — Amazon, Target, Walmart. Customer voice calls flow in, structured tickets flow out. Embed the call widget on your site, or hit the REST API directly. Authentication is a single bearer token.

Quickstart

  1. Copy your API key above.
  2. Make a test request to GET /tickets with the bearer header.
  3. Receive ticket data — or hook your callback to be notified when Reya files one.
curl 
# Confirm your key works — should list all tickets
curl -H "Authorization: Bearer YOUR_API_KEY" \
     $BASE_URL/tickets
node 
const resp = await fetch("BASE_URL" + "/tickets?priority=high", {
  headers: { Authorization: `Bearer ${process.env.REYA_API_KEY}` },
});
const { tickets, counts } = await resp.json();
console.log(`${counts.high} high-priority tickets`);

Authentication

Every request must include an Authorization header. Two modes are accepted:

  • Bearer token — for programmatic / server-to-server use. Recommended for integrations.
  • HTTP Basic — used by the browser-facing console UI. Don't use this from your backend.
http 
Authorization: Bearer YOUR_API_KEY

If auth fails, you'll get a 401 response with a JSON body explaining what's expected.

REST API

All endpoints return JSON. Errors return { "error": "message" } with a non-200 status.

Tickets

Tickets are the unit of work in Reya. A ticket can be auto-filed by Reya during a voice call, or created directly via API.

MethodPathDescription
GET/ticketsList tickets. Filters: ?status=, ?priority=, ?retailer=, ?q=.
GET/tickets/:idGet a single ticket plus its linked transcript.
POST/ticketsCreate a ticket from outside the voice flow.
PATCH/tickets/:idUpdate status / priority. Use to approve, deny, escalate, override.
POST/tickets/:id/replyAppend a human-agent reply or internal note to the audit log.
POST/tickets/:id/assistGenerate an AI-drafted reply using the transcript + customer context.
curl 
# Approve a ticket
curl -X PATCH $BASE_URL/tickets/RC-001 \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"status":"resolved","action":"approve","note":"Refund granted"}'

Sessions

Every voice call creates a session — and every session creates a ticket of kind: "call" the moment the user connects, even before Reya files anything.

MethodPathDescription
POST/sessions/startOpen a session manually. Body: { sessionId, retailer }.
POST/sessions/endClose the session. Body: { sessionId }.

Transcripts

MethodPathDescription
GET/transcripts/:sessionIdFull transcript for a session (every customer + Reya turn).
POST/transcripts/:sessionIdAppend a turn. Body: { role, text, retailer }.

Voice

If you embed the voice widget directly (instead of using the hosted UI), your client-side code uses these:

MethodPathDescription
GET/sessionMints a short-lived OpenAI Realtime ephemeral token + session config.
POST/retrieveRetailer-aware FAQ retrieval. Body: { retailer, question }.
GET/rubricThe scoring rubric used to derive ticket priority.

Embed the voice widget beta

Drop the iframe on any page you own. The widget renders the call button and streams audio directly between the browser and OpenAI — your server only handles session minting and ticket creation.

html 
<iframe
  src="BASE_URL/?token=YOUR_API_KEY"
  allow="microphone; autoplay"
  style="border:0;width:100%;height:560px"
></iframe>

For a tighter integration without an iframe, hit /session from your own UI, set up the WebRTC peer connection with the ephemeral token, and listen for tool-call events on the data channel. The hosted UI's public/index.html is the reference implementation — it's ~200 lines of vanilla JS.
